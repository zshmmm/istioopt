# "黑屏变更"操作流程规范

## 1. 背景

本操作流程变更规范旨在规范各种“人工变更”类的操作流程，在全部平台化、系统化之前，通过流程规范降低出故障的几率。

## 2. 范围

所有未纳入平台系统管控的生产环境变更或可能对生产环境有影响的人工变更操作。包括命令行变更操作、各云商平台操作、其他第三方Pass平台操作。这些操作有个共同点，需要人工进行判断。


## 3. 变更红线

### 3.1. 变更时间

  1. 周一~周四：上午10:00 - 11:30，下午14:00 - 17:30
  2. 周五及节假日封网前一天：上午10:00 - 11:30，下午：14:00 - 16:00

### 3.2. 通知

  1. 变更窗口期外：变更需邮件通知相关人员，并抄送SRE运维团队（sre.mail.com）和业务稳定性保障负责人（xx.mail.com）
  2. 变更窗口期内：变更前/后需在《业务变更通知》群中做变更和变更完成通知

### 3.3. 变更操作
  1. 明确变更时间
  2. 明确变更项操作流程
  3. 明确操作人，检查人
  4. 明确变更影响
  5. 明确变更后检查点
  6. 变更需要有明确的回退方案
  7. 预估回退时长

不满足变更红线的操作视为**违规操作**。如在非变更时间段变更需直属上级邮件审批后方可变更操作，变更时间段只需在《业务变更通知》群中同步即可。**变更影响较大且回退时长超过10分钟的，需要评估方案。**

## 4. 黑屏变更类型定义

### 4.1. 互联网入口切换操作
  1. 域名解析（公网DNS、HTTPDNS）
  2. 高防
  3. ELB
  4. IGTM
  5. CDN
  6. Nginx
### 4.2. 业务接入层变更操作
  1. Nginx
  2. global-proxy
  3. 业务网关
  4. 集群路由
  5. Istio-ingress-gateway
### 4.3. 存储层
  1. 数据库
  2. 中间件
### 4.4. 第三方Pass平台操作
  1. 云商平台（阿里云、腾讯云、华为云等）
  2. 第三方Pass平台（如：TAPD、Jenkins等）

## 5. 变更操作流程

```mermaid
flowchart TD
    A[发起需求] --> B{变更执行人评估}
    B -- 重要组件影响较大 --> C[发起相关依赖方联合评估风险]
    B -- 单一功能无需联合评审 --> D[形成变更方案]
    C --> D
    D --> E{负责人审批}
    E -- 驳回 --> B
    E -- 通过 --> F[执行人发起变更流程]
    F --> G{变更检查人检查}
    G -- 驳回 --> F
    G -- 通过 --> H[同步到变更通知群<br/>(重要依赖方需要当面确认)]
    H --> I{执行变更}
    I -- 变更失败 --> J[回滚]
    J --> K[变更群<br/>变更邮件<br/>需求发起人（业务）<br/>回顾失败原因并整改]
    K --> L[回顾失败原因并整改]
    I -- 变更成功 --> M[留观监控]
    M --> N[同步信息]
    N --> O[需求结束]
```

## 6. 风险评估内容

1. 变更人需要提前准备变更方案、风险预估、回滚预案、验收方法
2. 梳理变更关联方
3. 拉通关联方评估各自业务模块的风险
4. 确认变更演练/灰度测试/变更批次
5. 确认变更时间，以及通知方式（内/外部通告）
