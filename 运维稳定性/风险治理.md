# 风险治理

## 1. 背景

为了更好保障业务稳定运行，提前发现业务潜在风险，并及时治理。特制定此标准，用于规范日常业务的风险治理管控，建立“动态治理”的机制，最终实现线上环境风险相对可控。

**目标：风险治理的目标在于降低负面风险的发生概率和影响范围，从而提升业务的稳定性。**

## 2. 风险事件处理

解除危险事件主要有两种核心思路：

- 事前杜绝：从源头规避危险事件的发生
- 事后整改：在危险事件发生后，采取措施弥补损失、防止复发

## 3. 应对方式

针对来源不明、发生时间不确定，或随环境演进不断出现的新风险，通用应对步骤分为 3 步：
1. 建立风险库：基于历史经验，梳理并建立完整的风险清单，明确已知风险的核心信息；
2. 持续运营诊断：针对当前持续演进的环境，进行常态化、不间断的诊断与扫描，及时捕捉潜在风险；
3. 迭代更新风险库：发现新风险后，及时开展治理工作，同时将新风险总结抽象为通用风险项，纳入风险清单，实现风险扫描与诊断的常态化覆盖。

## 4 核心概念

### 4.1 动态

风险具有动态性，通过周期性巡检与治理，确保生产环境在一定周期内，将扫描到的风险按期完成治理，最终实现生产环境的相对安全可控。

### 4.2风险
指不确定的事件，或可能发生的、会对业务稳定性产生消极影响的各类事件，核心特征是不确定性与负面影响性。


### 4.3 治理
通过 “发现 — 分析 — 制定措施 — 落地 — 验收” 的闭环流程，在风险演变为故障前将其处置完毕，并将该风险归入风险扫描清单，纳入日常扫描与识别体系，形成长效防控机制。

### 4.4 风险三要素

1. 风险事件
2. 发生概率
3. 事件影响和后果

### 4.6 风险治理，检核范围

环境（流程规范、权限、资源、工具平台）、业务（程序代码、业务逻辑）、人员（组织机制、风险意识）

### 4.7 风险类型的定义

| 类型     | 描述                                                 | 举例                                                                                         | 备注 |
| -------- | ---------------------------------------------------- | -------------------------------------------------------------------------------------------- | ---- |
| 可靠     | 主要从部署角度去分析资源（服务）高可用情况           | 如是否单点部署，有没有跨可用区部署                                                           |      |
| 性能     | 响应资源（服务）的性能方面的指标去分析风险           | 检查 CPU 使用率情况，若使用率过高，可能会出现业务请求延迟增加，甚至无响应等风险               |      |

### 4.8 业务场景定义

业务场景是对具体业务或流程的一种分析方法，它主要聚焦于某一类业务或某一组流程在特定环境下的运行逻辑。

按照当前线上的运行情况，一般可以分为两类：
1. 固有场景，长期存在的，持续运营的统一场景：
- 横向的专业领域视角，比如统一数据库管理场景、统一接入层场景、统一容器集群场景等
- 纵向的业务垂直视角，比如一个完整应用，包含了应用入口，逻辑层，缓存层，数据层等

2. 临时场景，定时存在的，或短期持续一段时间进行的项目类：
- 业务活动，比如年度盛典、周年庆，等其他运营报备的大型活动
- 独立项目，比如云商迁移、统一集群改造、核心链路故障演练等

### 4.9 风险事件的定义

**风险：** 不确定的事件，或者可能发生的条件，会对业务的稳定性产生消极影响的事件都属于风险。

**风险事件的范式：** 某个业务场景下，某个检核对象可能存在的特定类型的，会对线上产生消极影响的事件。

## 5. 标准制定

### 5.1 风险等级定义

| 等级                 | 发生概率&&影响程度                     | 整改力度                                                                 | 风险升级机制       |
| -------------------- | -------------------------------------- | ------------------------------------------------------------------------ | ------------------ |
| 灾难级（类比：高风险） | 高发生概率且p1影响                      | 需要立即对线上进行熔断，停下其他工作，最高级别响应修复                   | 无                 |
| 严重级（类比：中风险） | 高发生概率且p2影响<br>中发生概率且p1影响<br>中发生概率且p2影响 | 在一定周期范围内，制定修复计划，有序完成整改                             | 存在时间超过XX时间 |
| 警告级（类比：低风险） | 低发生概率且p3以下影响                  | 当没有空余人力时可以暂时放缓，但随着风险存在的时间推移，风险会升级       | 存在时间超过XX时间 |

### 5.2 发生概率的定义

| 发生概率 | 概率描述                                                         |
| -------- | ---------------------------------------------------------------- |
| 高       | 随时可能发生的，没有任何门槛限制。比如线上所有人有核心路由修改权限，且存在组件零认证 |
| 中       | 只有线上变更才可能发生，比如每次扩容都会引起                     |
| 低       | 当没有空余人力时可以暂时放缓，但随着风险存在的时间推移，风险会升级 |

### 5.3 影响程度的定义

| 严重程度 | 后果                                                         |
| -------- | ------------------------------------------------------------ |
| p1       | 灾难级，出现既线上以上产生定级故障，比如线上已经存在的，持续对用户体验、流水收入或者业务数据造成影响的。 |
| p2       | 严重级，如果短时间（XX）内不处理，有大概率造成验证重大故障故障的风险隐患，比如线上权限机制存在漏洞，所有可以登录堡垒的同学都可以免密正删改核心链路的路由规则。 |
| p3       | 警告级，可能会造成线上影响的，比如分布式存储频繁发生节点剔除、数据库主备频繁倒换。 |

## 6. 风险治理流程

![风险治理流程][风险治理流程]

### 6.1 各环节细节

#### 6.1.1 发现

指通过某种手段感知和发现到线上风险，对风险定级，根据不同的级别响应风险。
发现风险可以通过人工巡检、风险管理平台自动扫描。
响应风险可以通过人工录入风险事件，以及风险管理平台基于扫描采集自动生成风险事件。

识别风险的工具和技术

1. 专家识别
邀请内部团队专家，或者外部领域专家，根据自身经验和专业知识，判定风险

2. 数据收集
核对单（checklist）
根据类型项目，或者历史项目积累的可能出现的问题和经验知识，编制的checklist

3. 头脑风暴
目标：获取全面的风险来源清单
参与人员：项目内成员、项目外专家
形式：采用自由、结构化（风险分析结构RBS）来开展，在引导者的指引下产生


4. 现状分析
发现分析导致问题的深层原因并制定预防措施，可用问题陈述、收集陈述作为出发点，使用5why法来发掘

#### 6.1.2 治理

治理主要分为两部分内容：
1. 根据发生概率，影响范围评估风险等级
2. 制定治理方案和计划
产生风险事件单后，启动的一系列治理管控流程，以及进度跟进督促的过程。
通过风险管理平台的事件单后续流转流程进行管理。

常见应对措施：

| 方式 | 定义 | 具体措施 | 说明 | 备注 |
| ---- | ---- | ---- | ---- | ---- |
| 上报 | 将风险向上汇报到被风险影响的层级 | 上报给消极风险影响到的层级 | 风险、威胁不在项目范围内，或者具体应对措施不在治理人员权限范围内，需要上报 |  |
| 规避 | 制定策略消除风险、威胁，或者保护项目免受风险影响 | 1、去掉有风险的任务<br>2、将受影响的项目目标从风险影响中分离出来，与第一点类似<br>3、调整受风险影响的目标<br>&nbsp;&nbsp;&nbsp;&nbsp;a、延迟交付时间<br>&nbsp;&nbsp;&nbsp;&nbsp;b、降低质量要求<br>&nbsp;&nbsp;&nbsp;&nbsp;c、调整优先级<br>&nbsp;&nbsp;&nbsp;&nbsp;d、调整项目范围和边界<br>4、放弃项目 | 适用于风险发生概率较高，且具有严重负面影响的高优先级风险 |  |
| 转移 | 将风险转移到第三方管理和承担后果 | 购买保险，内容外包，第三方担保 | 通常需要对外支付一定的风险费用 |  |
| 减轻 | 减轻负面影响发生的概率和影响范围 | 1、改变资源配置，比如调用更资深的人<br>2、减少流程，进行更多的测试，提高质量<br>3、使用更可靠的技术框架 | 提前采取减轻的措施，比事发后补偿更有效 |  |
| 接受 | 不主动采取措施，继续维持现状 | 1、主动接受，建立应急储备，预留时间、追加资源和资金，应对威胁的出现<br>2、被动接受，不采用对应措施，只是定期审查风险，是否出现重大变更 | 对于低优先级，或采用任何措施都无法改变的风险 |  |

#### 6.1.3 跟踪&验收

跟踪：根据制定的治理计划，定期追踪治理进度。
验收：分为非治理执行人的第三方人工验收（非相关同学），或者通过系统进行风险扫描。

#### 6.1.4 归档

关闭风险事件，数据归档到平台分析，风险验收通过后，会将该风险事件进行总结整理，然后关闭归档，为后续整个过程的统计分析提供数据。
如是新类型风险事件，需要进一步抽象成通用风险项，纳入风险识别清单。
不同类型的风险适用的治理方式

#### 6.1.5 复盘总结

风险事件的复盘总结，主要是对风险事件的发生原因、治理过程、结果进行总结分析，形成经验教训，为后续风险治理提供参考和指导。

#### 6.1.6 考核办法

风险治理的考核办法，主要是对风险治理的效果进行评估和考核，通常包括以下几个方面：

- 时效：治理闭环及时率

- 达标率：大型活动风险达标率，业务线应用风险达标率
